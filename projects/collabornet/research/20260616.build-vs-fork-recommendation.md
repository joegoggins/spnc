# Collabornet: build vs. fork — research summary

_Compiled 2026-06-16 from a multi-source, adversarially-verified deep-research pass (primary sources: project repos, READMEs, live GitHub/Codeberg API, cloned code, official docs). Companion to [build-vs-buy.md](build-vs-buy.md)._

## TL;DR

**Build fresh, minimal. Do not fork.** No existing open-source tool is simultaneously (a) map-first, (b) broad enough to cover assets + skills/time + surplus goods, and (c) multi-tenant-by-subdomain. Every candidate forces you to gut its core paradigm — slower than building the small thing you actually want.

The thing you want is small: **a multi-tenant CRUD-over-a-map**. Recommended stack: **Postgres + PostGIS, FastAPI, React + Leaflet, Zitadel for auth + tenancy** — with **GeoDjango as a faster-to-ship escape hatch** if pure calendar time beats stack-consistency.

The real risk is **adoption, not tech.** Ship the absolute minimum (map + pins + login) to one real MSP neighborhood before building multi-tenancy or anything else.

## 1. Landscape — why nothing is forkable

| Tool | Map-first? | Domain fit | Stack | License | Multi-tenant | Maintained | Verdict |
|------|-----------|-----------|-------|---------|--------------|------------|---------|
| **Karrot** | ❌ group/coordination-centric | mutual-aid (food-rescue origin) | Django (Py 97.7%) + Vue + PG + Redis | AGPL-3.0 | by **group**, not subdomain | ✅ commit 2026-06-04, v17.4.1, 3,088 commits | **Same "bloated, not map-first" problem as Hylo** |
| **Falling Fruit** | ✅ genuinely map-first | ❌ foraging only (1.6M edible-plant pins) | Rails 3 / Ruby 2.3.4 + PG15 **+ PostGIS** | GPL-3.0 | ❌ one global map | ⚠️ legacy "inactively maintained"; React successor `falling-fruit-web` active (push 2026-06-16) | Wrong domain, legacy fork base, no Docker |
| **TimeOverflow** | ❌ no map at all (code search "map" = 0 hits) | timebanking only | Rails 4.2 + PG | AGPL-3.0 | ✅ several banks/install (superadmin/admin/member) | ✅ commit 2026-04-10, v5.5.0 | Good tenancy model to *learn from*; not forkable |
| **Murmurations** | n/a — it's a **protocol**, not an app | regenerative/solidarity economy (exact fit) | Go microservices; protocol repo is just Markdown/Mermaid | GPL-3.0 | n/a | ⚠️ unfunded, volunteer-led | **Interoperability layer for later**, not a UI to fork |
| **Communecter** | map prominent but one of many tools | citizen/city/org/project/event (very broad) | PHP (Yii/Mongo) | — | — | GitHub mirror dormant (2019); **canonical GitLab active** (5,877 commits) | Broad/heavy → same Hylo trap |
| **Ushahidi** | ❌ map is one co-equal view (claim "map-first" **refuted** vs. its own docs) | crisis/data-collection, generic incident model | PHP 86.7% (Laravel) + **Angular** (not React) | AGPL-3.0 | — | ✅ v2025.04.0 | Wrong domain + non-React |

**Pattern:** the map-first tools (Falling Fruit, Ushahidi) are single-purpose / wrong-domain / legacy; the commons-aligned tools (Karrot, Communecter) are feed/group/governance-centric — *the exact thing you rejected in Hylo*; the clean multi-tenant tool (TimeOverflow) has no map. None is forkable without ripping out its center of gravity.

> Notable verified detail: **Karrot stores location as plain `latitude`/`longitude` FloatFields, not PostGIS** (cloned `karrot/base/base_models.py`), and its entities are groups → places → activities/offers/polls — confirming it is coordination-first, not "pin any resource on a map" first.

## 2. Recommendation: BUILD, minimal

Your data model is a weekend's worth of schema, not a platform:

```
collab_sites (id, name, slug, subdomain, center geography(POINT,4326), radius_m)
resources    (id, collab_site_id, kind ENUM[asset|skill_offer|skill_request|surplus],
              title, description_md, location geography(POINT,4326),
              created_by, created_at, status)
```

Three resource types collapse into **one `resources` table with a `kind` enum** — don't build three subsystems. Events/governance are out of scope (your call), which is exactly what keeps this small and is *why* Hylo/Karrot/Communecter feel bloated to you.

## 3. Recommended stack (verified viable)

- **Map: Leaflet.** ~42KB, raster tiles, simplest path to ship. MapLibre GL (WebGL/vector) is the upgrade *later* if you go offline/vector-heavy — not now.
- **Geo backend: PostgreSQL + PostGIS** (not plain lat/long — gives you the radius query for free).
  - Store points as the **`geography` type, SRID 4326** → distances are in **meters**, so "radius of stewardship" (100–500m, per [SPNC-0002](../stories/SPNC-0002.create-new-collab-site.md)) is expressed directly.
  - Hyper-local query uses **`ST_DWithin`** (uses the spatial index; faster + more accurate than `ST_Buffer`). This is the one geo primitive you need for v1.
  - `postgis/postgis:15-3.4-alpine` Docker image; `Geometry/Geography(..., spatial_index=True)` in the ORM.
- **Backend: FastAPI** (your lean) **+ SQLAlchemy 2 + GeoAlchemy 2**. Documented-viable combo (FastAPI + PostGIS + Leaflet/Geoman). With heavy AI-agent use, the boilerplate cost is acceptable.
  - **Escape hatch if ship-speed wins: GeoDjango + DRF-GIS.** Same Python family, but you get for free: `PointField` → auto `geometry(POINT,4326)` + GIST index; `GeoFeatureModelSerializer` (GeoJSON out); `InBBoxFilter` (map-viewport queries); and a **Django admin = instant CRUD for site setup + moderation**. That admin alone can shave days off the pilot. (It's literally Karrot's stack, minus the bloat.) Choose FastAPI to own every layer; choose GeoDjango to get neighbors using it a week sooner.
- **Auth + multi-tenancy: Zitadel** (your lean — verified strong fit):
  - **Each collab site = a Zitadel "organization"** (its native single-instance multi-tenant unit).
  - Subdomain → org routing via host-header / the `urn:zitadel:iam:org:id:{id}` scope; per-org branding + auth policy.
  - **RBAC** via Zitadel **projects → roles**, read from token scopes — there is an **official FastAPI example** (`github.com/zitadel/example-auth-fastapi`, Authlib + OIDC PKCE). Toggle "Return user roles during authentication" in the project.
  - Zitadel ships an **official Helm chart** + uses Postgres (fits your Helm/Helmfile plan) — but it targets 3 replicas and documents **no RPi/single-node minimums**, so on the Pi you'll likely run **hosted Zitadel Cloud** or a trimmed single instance.

## 4. Offline / resilience (the Meshcore + RPi angle — phase it in *later*)

- **Tiles offline: PMTiles** — entire tile pyramid in one file, served from static storage / **Cloudflare R2**, no tile server. BSD, active (~2.9k★), works with Leaflet *and* MapLibre. **10–100MB per archive is ideal for a hyper-local extent** — your 100–500m radius is tiny, so this fits perfectly.
- **RPi offline tile host: UNVT Portable** — RPi4 offline map server, CC0, active. A ready-made pattern for the Meshcore/grid-down scenario.
- **Full offline app:** PWA + Service Worker (to cache sprites/fonts) + PMTiles. Note the slick `maplibre-offline-pmtiles` (OPFS) plugin needs **MapLibre, not Leaflet**, and is very new/small (v2.1.1, ~15 commits) — don't depend on it yet.
- **Data sync:** ElectricSQL (self-host) / PowerSync (Postgres offline-first sync engines) exist for when you want device-local writes — but this is a Phase-2+ concern, not pilot.

---

## 5. Fastest path to a usable MSP pilot

- **Phase 0 — single-tenant slice (ship this to neighbors).** *Skip multi-tenancy entirely first.* One hardcoded collab site = your block. Map + pins (one `resources` table, `kind` enum, `ST_DWithin`) + Zitadel login. Raster OSM tiles. Deploy to a cheap ethical host. **This is the real milestone** — proves people will pin things.
- **Phase 1 — multi-tenant.** Add `collab_sites`, subdomain routing, Zitadel orgs + RBAC, and the admin "create collab site" flow ([SPNC-0002](../stories/SPNC-0002.create-new-collab-site.md)). Only build this once Phase 0 has real users.
- **Phase 2 — resilience.** PWA offline, PMTiles, RPi k3s, Meshcore tie-in.

**The bottleneck is neighbors actually pinning resources, not your stack.** Optimize Phase 0 for "least software between an idea and a pin on a map," and let adoption — not architecture — tell you whether Phase 1 is worth building.
