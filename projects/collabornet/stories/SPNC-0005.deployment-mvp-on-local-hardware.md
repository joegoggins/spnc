## Why

I own a NAS, several RPi 5s, and the domain `mspsolarpunk.com`. Hosting the site on hardware I
control is the Solarpunk thing to do. This story is the **fastest credible path to "hello
world" over legit https on my own metal** — and, just as important, it produces a **reusable
provisioning tool (`jk3s`)** so standing up the next box is one script + one `helmfile apply`,
not a weekend of hand-config.

The "hello world" is deliberately a **static smoke test, decoupled from the app**
([SPNC-0004](SPNC-0004.first-usable-local-thing.md) is still `todo`): it proves the whole
network path — DNS → Cloudflare edge → Tunnel → RPi → k3s → pod → https — independently of the
app being ready, so when the real app lands the platform is already trusted.

## How

### The `jk3s` project (the deliverable)

A standalone provisioning project at [`projects/jk3s`](../../jk3s/). **Scope of jk3s:**
provision *one* RPi 5 (NVMe/PCIe boot) into a trimmed single-node k3s cluster and deploy
cloudflared to serve a hello-world **https** endpoint at a given root domain. Running the same
tooling twice, parameterized, yields the dev and prod boxes:

| Box  | Hostname / kube-context / tunnel | Root domain        |
|------|----------------------------------|--------------------|
| dev  | `sp-dev`                         | `japoofis.com`     |
| prod | `sp-prod`                        | `mspsolarpunk.com` |

```
projects/jk3s/
  README.md                       # what jk3s is + end-to-end runbook (hardware -> https)
  bootstrap/
    bin/bootstrap.sh              # idempotent: fresh RPi5 -> trimmed single-node k3s
    README.md                     # prereqs (NVMe boot, EEPROM, cgroup, ASPM fallbacks), usage
  helmfile-base/
    helmfile.yaml.gotmpl          # environments: sp-dev, sp-prod; releases: cloudflared, hello-world (.gotmpl = templated, Helmfile v1)
    environments/
      sp-dev.yaml                 # rootDomain: japoofis.com,     tunnel: sp-dev
      sp-prod.yaml                # rootDomain: mspsolarpunk.com,  tunnel: sp-prod
    charts/
      cloudflared/                # local chart: Deployment + config.yaml (ingress -> hello-world Svc)
      hello-world/                # local chart: Deployment + Service + ConfigMap ("hello world")
    secrets/                      # tunnel credentials JSON, CLEARLY MARKED (see repo secret convention)
```
(Extra subdirs are fine if they earn their place — e.g. a `bin/` kubeconfig-fetch helper.)

### Step 1 — node provisioning (`bootstrap/bin/bootstrap.sh <cluster-name>`)

Idempotent script, run **on the RPi** after it's NVMe-booting a fresh 64-bit OS. It:
- ensures the cgroup memory flags are present in `/boot/firmware/cmdline.txt`
  (`cgroup_memory=1 cgroup_enable=memory`) — on Pi 5 / Bookworm verify they actually take effect
  (the DTB has historically re-disabled the memory cgroup); reboots if it changed them;
- installs k3s **trimmed for this use** — Cloudflare Tunnel is the ingress, so no in-cluster
  LB/ingress is needed:
  `INSTALL_K3S_EXEC="--disable traefik --disable servicelb --write-kubeconfig-mode 644"`
  (**local-path is kept** — it's the default StorageClass that backs PVCs on the NVMe);
- names the node `<cluster-name>` and emits a kubeconfig whose context is `<cluster-name>` so
  the laptop can `kubectl`/`helmfile` against `sp-dev` / `sp-prod` unambiguously.

### Step 2 — Cloudflare Tunnel (one-time per box, then in git)

Locally-managed tunnel so hostname→service routing lives in the repo, not the dashboard:
1. `cloudflared tunnel create sp-dev` → produces a **credentials JSON** (the secret).
2. `cloudflared tunnel route dns sp-dev japoofis.com` → CNAME (apex via flattening).
3. credentials JSON lands in `helmfile-base/secrets/` (clearly marked per repo convention).

### Step 3 — deploy (`helmfile -e sp-dev apply`)

`helmfile-base` deploys, in order:
- **cloudflared** — the **first and only ingress** into the cluster. Runs as a Deployment with
  a `config.yaml` ConfigMap whose ingress rule maps `rootDomain → http://hello-world.<ns>.svc:80`,
  authenticated by the tunnel credentials Secret. No traefik, no LoadBalancer.
- **hello-world** — a tiny Deployment + ClusterIP Service serving a "hello world" page from a
  ConfigMap. cloudflared reaches it by cluster DNS.

Then `https://japoofis.com` shows "hello world", TLS terminated at Cloudflare's edge (free,
auto-renewing). Re-run with `-e sp-prod` for `mspsolarpunk.com`.

## Hardware notes — Pi 5 NVMe gotchas (canonical copy lives in `jk3s/bootstrap/README.md`)

The order these tend to bite, with fixes:

1. **PCIe power-state / ASPM** — by far the most common "drive not detected / I/O errors" cause,
   and more likely on DRAM-less drives like the TN320. Escalating fixes (forum-documented):
   - `pcie_aspm=off` in **cmdline.txt** (not config.txt) — note this raises board power draw,
     it also de-tunes the RP1 link;
   - add `nvme_core.default_ps_max_latency_us=0 pcie_port_pm=off`;
   - `dtparam=pciex1_no_l0s` in config.txt for drives with a broken L0s;
   - last resort: drop to `dtparam=pciex1_gen=1`.
   Probably won't be needed — but keep this list; it saves an evening when a node won't see its disk.
2. **Power supply** — use the official 27W USB-C PD supply. The Pi 5 throttles peripheral power
   without it, and the M.2 socket's 3.3V comes off a marginal DC/DC converter on weak supplies.
3. **Detection basics** if `lsblk` is empty: reseat the FPC cable at both ends (locking tabs must
   click), check orientation, confirm the drive is M-key NVMe (it is). Easy to ribbon it backwards.

Enable PCIe in `/boot/firmware/config.txt`: `dtparam=pciex1` (alias `dtparam=nvme`); optional
`dtparam=pciex1_gen=3` for ~2× throughput (unofficial/may be unstable — Gen2 is the safe default).
Enable NVMe boot via `raspi-config` → Advanced → Boot Order → NVMe/USB (or `BOOT_ORDER=0xf416`
via `rpi-eeprom-config`; a third-party HAT may also need `PCIE_PROBE=1`).

## Done when

**Core acceptance (the original intent):**
- [ ] Hello-world reachable over **legit https** at the dev root domain (`https://japoofis.com`).
- [ ] Hello-world reachable over **vanilla http** (note: Cloudflare "Always Use HTTPS" redirects
      http→https by default; toggle off if a literal plain-http response is wanted).
- [ ] Same on **prod**: `https://mspsolarpunk.com` says hello world.

**Provisioning / reproducibility acceptance (the jk3s payoff):**
- [ ] `bootstrap/bin/bootstrap.sh` turns a fresh NVMe-booted RPi 5 into a trimmed single-node k3s
      (traefik + servicelb disabled, local-path retained), idempotently, with a `<cluster-name>`
      kube-context.
- [ ] `helmfile -e <env> apply` deploys cloudflared + hello-world; cloudflared is the **only**
      ingress path (no traefik, no port-forwarding, no inbound firewall holes).
- [ ] **Prod is stood up by re-running the same tooling** with `-e sp-prod` — no hand-editing,
      no drift between boxes.
- [ ] `projects/jk3s/README.md` documents the full path: hardware/NVMe boot → bootstrap →
      tunnel create → helmfile apply → live https.

## Notes & decisions

- 2026-06-30 — **k3s over Talos.** Talos fits the immutable/declarative ethos, but Pi 5 support
  is community-grade (custom image build, version/stepping-sensitive boot regressions, NVMe-boot
  rough edges) — it makes *provisioning* the hard part. k3s is one `curl` and rock-solid on Pi 5.
  Revisitable later as a platform swap; k8s workloads are portable either way.
- 2026-06-30 — **Trim k3s:** `--disable traefik --disable servicelb`, keep local-path. Cloudflare
  Tunnel is the ingress; no in-cluster LB/ingress needed.
- 2026-06-30 — **cloudflared is the first + only ingress**, deployed in-cluster. **Locally-managed
  tunnel** (config + credentials in `helmfile-base`) so hostname routing is in git, not the
  dashboard. Revisable to a token/dashboard-managed tunnel if that's simpler to operate.
- 2026-06-30 — **NVMe/PCIe boot from day one** (2× 52Pi N10 + Inland TN320 256GB). SD card kept
  only to bootstrap NVMe boot. DRAM-less/HMB drives can hit Pi 5 PCIe ASPM quirks — fallbacks above.
- 2026-06-30 — **PVCs from local-path on the NVMe**; NAS becomes the backup target later.
- 2026-06-30 — **Naming:** `sp-dev`/`sp-prod` is simultaneously the node hostname, the kube-context,
  and the Cloudflare tunnel name, per box.
